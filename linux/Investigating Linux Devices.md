## auditd

Learn about the Linux Auditing System and the powerful logging capabilities provided by auditd.

**Resources:**  
[The Linux Audit Documentation Project](https://github.com/linux-audit/audit-documentation)

### Audit Rule for Monitoring File Operations:

Command: 
```bash
$ sudo auditctl -w /etc/passwd -p wa -k passwd_changes
```

Explanation: Monitors write and append operations for /etc/passwd, crucial for tracking potential unauthorized changes

### Audit Rule for Monitoring Program Execution System Calls:

Command: 
```bash
$ sudo auditctl -a always,exit -F arch=b64 -S execve -k exec_tracking
```

Explanation: Captures program execution events, aiding in identifying unexpected or malicious activity

### Audit Rule for Tracking Network Activity:

Command: 
```bash
$ sudo auditctl -a always,exit -F arch=b64 -S connect,bins -k network_activity
```

Explanation: Essential for investigating suspicious network bindings and connections

### Proxmox Memory and Disk Collection

#### Memory Collection

Memory can be extracted from both running and paused systems on the command line using the following steps:

"qm monitor [ID]" (e.g., "qm monitor 101")

"dump-guest-memory -d /tmp/[dumpname].mem" (e.g., "dump-guest-memory -d /tmp/101-running.mem")

"help dump-guest-memory" to get options, "quit" to exit

You can also hibernate ("suspend to disk"). This obviously prevents dump-guest-memory from working. A [VMID]-state-suspended-[DATE].raw file can be found in /dev/pve/. This appears to be a QCOW suspended disk state file (QEVM header) that contains volatile memory as well as other data.

Memory can be extracted with Volatility 3's LayerWriter plugin

Get a list of layers with "vol.py -f vm-101-state-suspend-2024-04-08.raw  layerwriter.LayerWriter --list"

Extract the memory layer with "vol.py -f vm-101-state-suspend-2024-04-08.raw  laerwriter.LayerWriter --layers memory_layer"

In testing, this resulted in a "memory_layer.raw" file that was appx 2GB larger than the allocated memory (6 GB vs 4.2 GB with dump-guest-memory, host had 4GB assigned).

Given the extra space utilization, as well as the need to convert from an intermediary file, there may not be any benefit to this method.

#### Disk Collection

Disks are probably (depending on setup/config) stored as block devices, not actual files on the file system

Find the primary disk name with "qm config [VMID]" (e.g., "qm config 101 | grep disk")

Disks should be in /dev/pve/, but you can use "find" to locate it once you know the name

Copy the block device to a physical file: "dd if=/dev/pve/vm-101-disk-1 of=/tmp/vm-101-disk-1.raw"

Reference: "VM disks aren't files on top of a filesystem but are block devices (zvols or LVs) if you are using ZFS or LVM/LVM-thin as the storage. So there are no files you can see or copy. Blockdevices you can only work with when using the CLI commands (for example lvs or zfs list) or when working with them using the "/dev/..." paths (like working with real physical disks)." https://forum.proxmox.com/threads/vm-disk-location-cant-find-the-vm-disk.101118/

Or move/convert the disk to a non-block device format (e.g., place it on a NAS as a QCOW or VMDK image)

## Installing and Configuring Volatility 3

Learn how to install and configure Volatility 3 and build the necessary Intermediate Symbol Format (ISF) JSON file necessary to analyze a Linux memory image.

Note that we'll be using dwarf2json to generate debug symbols tailored to the specific Linux kernel version in our memory image. This method enhances accuracy by ensuring an exact match of debug symbols, critical for reliable analysis with Volatility. Standard Linux kernels often lack necessary debugging information, so we opt for this approach rather than depending on potentially mismatched symbols from public servers.

**ℹ️ Tip:**  
While not used in this course, note that **Volatility 3** offers a `--save-config` option and a `-r pretty` formatting option that may be useful to you:

- The former option will allow you to save metadata for the memory image to speed up subsequent execution of plugins (recall the saved config with `-c`).
    
- The latter option will provide a cleaner output for many of the plugins to make it easier to interpret the results.

```shell
# Volatility 3 Installation (*** Ubuntu 22.04 ***)

# Create and Change Into Tools Directory
mkdir ~/tools
cd ~/tools

# Install Volatility 3
sudo apt update
sudo apt install -y python3-pip python3-pefile python3-yara git
pip3 install pycrypto
git clone https://github.com/volatilityfoundation/volatility3.git
cd volatility3

# Verify Successful Installation
python3 vol.py -h | more

# Determine Linux Kernel Version via Volatility 3 "banners" Plugin
python3 vol.py -f /path/to/image banners

# Install Debug Symbols
# This will vary depending on the kernel version used by the system from which the memory was acquired.
# For yamato.lime, we’re using 6.5.0-26.

sudo apt install ubuntu-dbgsym-keyring

echo "deb http://ddebs.ubuntu.com $(lsb_release -cs) main restricted universe multiverse
deb http://ddebs.ubuntu.com $(lsb_release -cs)-updates main restricted universe multiverse
deb http://ddebs.ubuntu.com $(lsb_release -cs)-proposed main restricted universe multiverse" | \
sudo tee -a /etc/apt/sources.list.d/ddebs.list

sudo apt update
sudo apt install linux-image-6.5.0-26-generic-dbgsym

# Install dwarf2json
cd ~/tools
git clone https://github.com/volatilityfoundation/dwarf2json.git
cd dwarf2json
sudo apt install golang-go
go build

# Use dwarf2json to Create Volatility 3 Intermediate Symbol File [ISF] JSON Output
./dwarf2json linux --elf /usr/lib/debug/boot/vmlinux-6.5.0-26-generic > output.json

# Move JSON Output to Volatility 3 Symbols
mv output.json ../volatility3/volatility3/symbols

# Verify Success
cd ~/tools/volatility3
python3 vol.py -f /path/to/image linux.pslist

```
